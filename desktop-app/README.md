# Atyrau Armsport — Desktop, Этап 6: модуль sync/

`armwrestling_tournament.py` — ваш исходный файл. Изменено ровно два места
(оба — чисто аддитивные, ни одна существующая функция не переписана):

1. **Один блок** сразу после `Database.close()` — оборачивает 5 методов
   (`create_tournament`, `add_category`, `add_participant`,
   `update_participant`, `save_match`): сначала выполняется оригинальный
   метод как раньше, потом результат уходит в центральную БД через
   `sync/`. Ни строки в самих методах не изменено — обёртка снаружи.
2. **Одна кнопка** «📡 Опубликовать результаты» рядом с «🎫 Печать
   бейджиков» на вкладке «Участники» + один метод-обработчик
   `_publish_tournament`.

Движок сетки (`DoubleEliminationEngine`), UI, генератор бейджей/PDF —
не тронуты вообще.

## Что делает модуль sync/

```
sync/
├── config.py          # API_BASE_URL, токен — из sync_config.json или env
├── state.py            # sync_state.db: карта локальных id <-> центральных
│                        # + офлайн-очередь (НЕ трогает armwrestling.db)
├── api_client.py        # тонкий HTTP-клиент к /api/v1/sync/*
└── sync_manager.py       # оркестрация: on_tournament_created,
                           # on_participant_added (ищет/создаёт спортсмена
                           # в центральной базе), on_match_created/updated,
                           # publish_tournament, flush_pending
```

**Модель:** действия организатора пишутся в центральную PostgreSQL сразу,
по мере того как происходят (турнир, категория, участник, каждый матч) —
не пакетом в конце (см. ARCHITECTURE.md §0/§5). Кнопка «Опубликовать
результаты» лишь переключает статус турнира `draft → published`, то есть
делает его видимым на публичном сайте.

**Если нет сети** — каждое действие всё равно выполняется локально без
задержек (табло/судьи не почувствуют разницы), а несинхронизированная
операция уходит в офлайн-очередь `sync_state.db` и повторяется через
`sync_manager.flush_pending()` (вызывается автоматически при попытке
опубликовать результаты, если в очереди что-то есть).

## Настройка

Создайте `sync_config.json` рядом с `armwrestling_tournament.py` (или
переменные окружения с теми же именами):

```json
{
  "API_BASE_URL": "http://localhost:8000/api/v1/sync",
  "DESKTOP_SYNC_TOKEN": "change-me-desktop-sync-token"
}
```

## Известное ограничение, обнаруженное на этом этапе

Текущая форма участника в десктопе **не собирает пол** (только ФИО, вес,
клуб, категория, рука, фото). Центральная `athletes.gender` теперь
nullable (миграция `861d708b8a1a`) — синхронизация не падает, но
профиль на сайте до дозаполнения пола не попадёт в фильтры "Мужчины"/
"Женщины". Рекомендация на будущее: добавить необязательное поле "пол" в
диалог `_add_participant_dialog` (некритично для Этапа 6 — не блокирует
sync, просто поле будет пустым, пока кто-то не заполнит его через сайт).

## Проверено вживую (реальный Database-класс, реальный backend)

| Сценарий | Результат |
|---|---|
| `create_tournament` → `add_category` → `add_participant` ×2 → `save_match` (создание + обновление победителя) | все 5 операций синхронизировались в центральную Postgres без единой ошибки; карта id заполнена (`competition 1 → 35`, `category 1 → 35`, `participant 1/2 → 35/36`, `match 1 → 35`) |
| Спортсмены видны в `GET /public/athletes` **сразу**, пока турнир ещё `draft` | подтверждено — именно то, что требовалось (п.1/2 диалога) |
| `sync_manager.publish_tournament(tid)` | турнир появляется в `GET /public/competitions?status=published` |
| API недоступен (порт закрыт) во время `create_tournament`/`add_category`/`add_participant` | все 3 операции всё равно выполнились ЛОКАЛЬНО мгновенно, ушли в офлайн-очередь (`pending_count() == 3`) |
| `flush_pending()` после восстановления сети | все 3 операции доехали в правильном порядке зависимостей (турнир → категория → участник), очередь опустела, спортсмен появился на сайте |
